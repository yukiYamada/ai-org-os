# AI主導開発組織 MVP Issueセット（2026-05-21）

## Epic
**AI主導で自立的に動く開発組織（人間最終承認あり）のMVPを確立する**

---

## Issue 1: Guardrails（やってはいけないこと）を定義する

### 背景
AI主導運用では、禁止事項・承認境界が曖昧だと事故と手戻りが増える。

### ゴール
禁止事項、承認必須条件、迷った時の既定動作を明文化する。

### 非ゴール
- 実行エンジン実装
- 監視/通知の自動化

### 受け入れ条件
- [ ] 禁止事項が5〜10項目定義されている
- [ ] 人間承認が必要な条件が定義されている
- [ ] 迷った時の既定動作（停止→代替案提示→人間承認待ち）が定義されている
- [ ] 影響範囲（org/team/project）が記載されている

### タスク分解
- [ ] T1-1 現行ルールから禁止事項候補を抽出
- [ ] T1-2 承認境界（破壊的変更・外部公開・依存追加など）を整理
- [ ] T1-3 文書化し、レビュー観点を定義

## Issue 2: Roles（やるひと）を定義する

### 背景
「誰が何を決めるか」が曖昧だと責務混在し、品質が落ちる。

### ゴール
Planner / Builder / Reviewer の責務、入出力、ハンドオフを定義する。

### 非ゴール
- マルチエージェント実行基盤の導入
- ロール別モデル最適化

### 受け入れ条件
- [ ] 各ロールの責務が1ページ以内で説明されている
- [ ] 入力/出力フォーマットが定義されている
- [ ] ハンドオフ手順が定義されている
- [ ] 人間の最終承認ポイント（マージ可否/公開可否）が明記されている

### タスク分解
- [ ] T2-1 ロール責務ドラフト作成
- [ ] T2-2 入出力テンプレート作成
- [ ] T2-3 ハンドオフ手順と例外時フローを定義

---

## Issue 3: Workspace（作業場所）を定義する

### 背景
作業場所と運用手順が不定だと、再現性と移植性が失われる。

### ゴール
LocalPC + Git を前提に、最小運用フローを確立する。

### 非ゴール
- Docker/Compose 導入
- クラウド常時稼働

### 受け入れ条件
- [ ] ローカル実行手順が3ステップ以内で記載されている
- [ ] ブランチ/PR/Merge の運用が定義されている
- [ ] 1 Issue = 1 目的 = 1 PR の原則が明記されている
- [ ] 将来の Organization 配下 repo 分割方針が記載されている

### タスク分解
- [ ] T3-1 ローカル運用手順を定義
- [ ] T3-2 Git運用ルールを定義
- [ ] T3-3 将来のrepo分割原則を定義

---

## Issue 4: Research Loop（情報収集）を定義する

### 背景
自立改善には外部情報収集が必要。ただし無秩序な収集は品質を下げる。

### ゴール
情報収集のトリガー、情報源優先度、記録方式を定義する。

### 非ゴール
- スクレイピング自動化
- ナレッジベースの大規模構築

### 受け入れ条件
- [ ] 収集開始条件（いつ調べるか）が定義されている
- [ ] 情報源の優先順位（一次情報優先）が定義されている
- [ ] 記録テンプレ（source/date/summary/impact）が定義されている
- [ ] 収集結果を `decisions.md` / `learnings.md` に振り分ける判定ルールが定義されている
- [ ] 矛盾情報の扱い（比較記録して保留）が定義されている

### タスク分解
- [ ] T4-1 情報収集ルール草案作成
- [ ] T4-2 記録テンプレ作成
- [ ] T4-3 運用サイクル（週次見直し）を定義
