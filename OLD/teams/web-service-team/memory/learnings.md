## L-0001: SSOT を先に固定する整合バッチの有効性

### 文脈
- 元となった出来事: PR #11 Vision 整合バッチ。末端ドキュメント群が `docs/product_vision.md` の3軸思想と段階的にズレており、個別発見ごとの漸進修正では再発が止まらなかった。
- いつ: 2026-05-22

### 学び（再利用可能な原則）
- 末端ドキュメント群が SSOT とズレている場合、個別漸進修正より「SSOT 先固定 → 並列で末端整合」のほうが速く再発も少ない。
- SSOT 整合は1 PR に集中させると単一目的を維持でき、レビュー観点の混線が起きない。

### 適用条件
- 影響を受けるファイルが概ね 10 個以上
- SSOT が未確立 or 古い（Vision が末端より新しいケース）
- サブエージェント並列実行が可能
- 1サイクル内に「SSOT 確立 → 末端整合 → 検証」を完結できる時間がある

### 反映先候補
- `org/operating_principles.md` には反映済み（6. Vision-First）
- `CONTRIBUTING.md` のチェックリストへも反映済み
- 次回 L1 再スナップショット時に L1 内 operating_principles へ伝搬予定

## L-0002: snapshot model の判断軸（live sync / canonical / snapshot の比較）

### 文脈
- 元となった出来事: D-0003 採択。L1 バンドル v0.1 で実体収録する際、本体（`teams/`, `templates/`）との二重管理を解く必要があった。
- いつ: 2026-05-22

### 学び（再利用可能な原則）
- 「商品としてのバンドル」と「ドッグフーディング用の本体」を両立させたい場合、live sync は中間状態を商品に流出させ、bundle-as-canonical はドッグフーディングの可読性を毀損する。snapshot model（節目で凍結スナップショットを取る）が両立解となる。
- snapshot model を採るなら、**起源↔スナップショットの乖離検出を学習ループ（retrospective / Metrics Review）に組み込む**ことが運用条件。組み込みがなければ snapshot は腐る。

### 適用条件
- 商品としての再現性・凍結性が求められる成果物がある（バンドル / リリースアーティファクト）
- かつ、起源側を継続的に進化させたい（本体 / 開発リポジトリ）
- 起源↔成果物の diff を定期検出する運用枠が用意できる

### 反映先候補
- `teams/web-service-team/memory/decisions.md` D-0003 に明文化済
- `templates/team/memory/retrospectives.md` に「起源↔バンドル乖離チェック」を必須項目として追加済
- 次回 L1 再スナップショット時に snapshot 運用 README をバンドルにも収録

## L-0003: L1 が空のまま L2/L3 を肥大化させる偏りの検出法

### 文脈
- 元となった出来事: Vision 整合作業中に、L1（商品本体）が骨子のみで実体ファイルゼロ、一方 L2/L3 と本体 `teams/` が大量に進化している構造的偏りが顕在化した。
- いつ: 2026-05-22

### 学び（再利用可能な原則）
- 商品本体（L1）/ 基盤（L2）/ 学習記録（L3）の3層構造を持つプロダクトでは、「商品本体の中身ゼロ」が最も気付きにくい欠陥になる。L2/L3 はドキュメント量が多いため、外形上は健全に見えてしまう。
- 検出指標: 「L1 配下の実体ファイル数」「L1 配下の最終更新日 vs L2/L3 の最終更新日」「L1 への参照リンク数」の3点を Metrics で観測すれば偏りに早期に気付ける。
- 是正策: L1 充填を最優先タスクとして backlog の最上段に固定し、L2/L3 の新規拡張に着手する前に L1 への寄与を必ず問う。

### 適用条件
- 多層構造（商品本体 / 基盤 / 学習記録）を持つドキュメンタリープロダクト全般
- 開発者が同時にユーザーでもある（ドッグフーディング）プロダクトで特に有効

### 反映先候補
- `projects/ai-org-os/backlog.md` 最上段に「L1 バンドル充填」固定済
- `projects/ai-org-os/specs/metrics.md` への指標追加候補（L1 実体ファイル数 / L1 参照リンク数）

## L-0004: カスタマイズマーカー（Must/Should/May）が受領者の認知負荷を下げる

### 文脈
- 元となった出来事: PR #13 で L1 v0.2 にカスタマイズガイドを追加し、各ファイルのどこを Must で書き換え、何が Should、何が May かを明示した。
- いつ: 2026-05-22

### 学び（再利用可能な原則）
- 配布物（バンドル / テンプレート）に対し、受領者は「どこから手をつけるか」で最も詰まる。3段階のカスタマイズマーカー（Must=必ず書き換え / Should=推奨書き換え / May=任意）を導入すると、最初の意思決定速度が大きく上がる。
- Must は固有名詞・組織名・スコープなど「そのままだと事実誤認になる箇所」に限定すること。Must を増やしすぎると逆に立ち上げ速度が落ちる。

### 適用条件
- 配布される成果物（テンプレート / バンドル / スターターキット）
- 受領者の前提知識がばらつく（社外配布 / 新規プロジェクト適用）

### 反映先候補
- `L1_product-os-bundles/web-service-team-bundle-v0/install.md`（v0.2 で実装済）
- 次回 L1 再スナップショット時に他バンドルにも展開
- `templates/` 配下にも段階的にマーカーを導入検討
