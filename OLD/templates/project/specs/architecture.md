## Architecture Template

> 想定読者: このプロジェクトの実装者（AI/人間）と、設計をレビューする意思決定者・テックリード。
> 「何を作るか」は requirements.md、ここは「**どう作るか**」を書く。

## 方針
<!-- 設計の指針を 3〜5 行で。技術選定の前提となる価値判断（例: 「依存最小・移植容易」「ベンダーロック回避」「学習コスト最小」）を書く。Vision の 3 軸との整合（特に Share = 移植可能性）に必ず触れる。 -->

## 構成
<!-- システム構成を箇条書きまたは図で。各構成要素について「責務」「主要技術」「他要素との依存方向」を書く。
例:
- **frontend**: Next.js / 画面提供 / backend に HTTP で依存
- **backend**: Node.js + Express / API 提供 / DB に依存
- **db**: PostgreSQL / 永続化 / 他に依存なし
-->

## データフロー
<!-- ユーザー操作 → どの構成要素を経由 → どこに永続化されるか、を 1 シナリオ書く。複数あるなら主要シナリオ 1 つで十分。 -->

## 検討事項
<!-- 未決事項・トレードオフ・ADR 候補。決まったら別ファイル（decisions.md 等）へ退避してよい。
例:
- 認証方式（自前 / Auth0 / Clerk）→ コスト vs 学習コストで未決
- 非同期処理基盤（Cron / Queue）→ スケール要件確定後に決定
-->
