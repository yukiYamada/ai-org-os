# ADR-0001: ai-org-os を「開発組織の不変項」を定義するフレームワークとして再定義する

> 想定読者: ai-org-os の方向性を理解しようとする全員（コントリビュータ、利用者、未来の自分）。

## Status

**Accepted** — 2026-05-22

## Context（背景）

### これまでの経緯

ai-org-os は当初、「自社開発 AI チームを構築・共有可能にするツール」と定義され、4 PR を経て以下を整備した。

- PR #11: Vision 整合バッチ（`docs/product_vision.md` を SSOT として確立、25 ファイル整合化）
- PR #12: L1-V0.1（`web-service-team-bundle-v0` 実体ファイル収録、snapshot model 採用 = D-0003）
- PR #13: 製品成熟化バッチ（L1 v0.2 カスタマイズガイド / README Quick Start / Metrics SSOT）
- PR #14: Round 3（テンプレ実体化 / Roadmap+FAQ / L3 学習ループ起動）

「製品としてまともな状態」を追求する過程で、リリース基準と Issue 一覧を整理しようとした矢先、根本的なズレに気づいた。

### 壁打ちセッションで露わになったズレ（2026-05-22 同日）

ユーザー（プロダクトオーナー）との壁打ちで、以下が判明した。

1. **「組織」のメンタルモデルは本質ではなく派生**
   - 既存文書群（charter / roles / workflow / rules）は「具体的な組織骨格」を記述している
   - だが本体は「**組織骨格を定義できる能力**」であって「組織骨格そのもの」ではない

2. **web-service-team-bundle は商品ではなく実装例**
   - 「L1 = 商品」と従来位置づけてきたが、誤り
   - 本体は「**フレームワーク**」であり、バンドルはそれを使った 1 つのサンプル

3. **「人間最終承認」表現が誤誘導**
   - 実態は「**事前定義された承認境界 + 内側は AI が委任で自走**」
   - ユーザーと Claude（私）の関係そのもの：自由にやってはいけないラインだけ事前に決め、内側は任せる

4. **「人間」というカテゴリは組織内に存在しない**
   - 人を分解すると「思考」と「物理デバイス（体）」になる
   - 組織に必要なのは**思考のみ**
   - 思考の実装が AI でも人間の脳でもよい（過渡期には人間脳実装が混在）
   - 「人 = 組織内のメンバー」ではない。「人 = ある思考の現在の実装媒体」

5. **「自立」の定義もズレていた**
   - 旧: 「人間が最終承認しつつ AI 主導で動く」（受動的・毎回承認）
   - 新: 「事前定義された承認境界の範囲内で AI が自走する」（能動的・境界委任）

### 5 年後のポジション

ユーザーの想定: 「これがスタンダードか、類似の大手が作ったものになるはず」。
つまり ai-org-os は **AI 駆動開発組織のデファクトスタンダードを目指す層**にいる。

---

## Decision（決定）

### 1. 本体の再定義

> **ai-org-os は、「開発組織とは、どんな思考が揃った集合か」の不変項（公理系）を定義するフレームワークである。各思考の実装（AI / 人間の脳 / 将来的なデバイス）は問わない。**

比喩: **公理系**。ユークリッド幾何学の5公理のように、組織はその上に独自の世界を組み立てるが、公理を否定したら別物になる。

並列の比喩: **Claude Code の組織版**。Claude Code が AI エージェント単体のハーネス（CLAUDE.md / skills / hooks / settings.json）を定義するのと同じレイヤーで、組織単位の動作を定義する。

### 2. 組織のモデル

**組織 = 思考のネットワーク**。

- 各「思考」は何らかの実装を持つ（AI / 人間の脳 / 将来的なデバイス）
- 実装は実行基盤であって、組織の構成要素ではない
- 「人間」というカテゴリは組織内に**存在しない**
- 過渡期に人間脳で実装される思考があっても、構造的には他の思考と同列

### 3. 不変項の積み上げ（試案、要さらなる練り込み）

「開発組織」を**思考**から積み上げる。各 Level の条件を満たさないと「開発組織」ではない。

| Level | 条件 | これがないと |
|---|---|---|
| **0** | 思考が1つ以上存在する（実装問わず） | そもそも何もない |
| **1** | 思考が **目的** を持ち、**記憶** を残し、**検証可能** な出力を生む | 「思いつきで何かする存在」止まり |
| **2** | 思考に **境界**（やってはいけないライン）が事前定義されている | 暴走 / 共存不能 |
| **3** | 思考が **自己更新（学習）** できる | 成長しない |
| **4** | 思考が複数のとき、**役割** で責任分担される | 個体の集合であって組織ではない |

**開発組織 = Level 0〜4 すべて揃った思考のネットワーク**。

### 4. 既存成果物の取り扱い

- 既存全成果物を `OLD/` 配下へアーカイブする
- 新規再構築はこの ADR を起点として、不変項フレームワークの定義から始める
- 既存資産（`OLD/` 配下）は参照可能だが、新方向と整合しない部分は捨てる前提

### 5. リファレンス実装の扱い

- 当面、リファレンス実装（web-service-team-bundle 相当）は**作らない**
- 本体（不変項定義）が固まってから、必要に応じて作る
- 例があると本体の抽象度が下がる懸念があるため、慎重に判断する

## Consequences（影響）

### ポジティブ

- 本体が抽象化され、特定ドメイン（web-service）に縛られない
- 5 年後のスタンダード / 大手後追いポジションに構造的に整合する
- 「人間最終承認」表現の誤誘導が解消され、AI 自走の本質を表現できる
- 「組織」を実装非依存に定義できる（AI / 人間脳 / 未来のデバイスを問わない）
- 不変項アプローチにより、組織が満たすべき条件が明文化できる

### ネガティブ

- 既存成果物の大規模アーカイブ（`OLD/` 配下に 10+ ディレクトリ）
- 過去の Vision / charter / specs / バンドル等の整合が一旦リセットされる
- 再構築コストが発生
- 一時的に「使える成果物」がなくなる（フレームワーク本体が出来上がるまで）

### リスク

- 抽象度が上がりすぎて、実利用時に「結局何を書けばいいか」がわからなくなる
- 不変項の定義（Level 0〜4 の試案）が間違っており、再壁打ちが必要になる
- 「人間というカテゴリは組織内に存在しない」原則が、過渡期の現実と乖離する可能性

### 対応方針

- 不変項の試案（Level 0〜4）は今後の検討で必ず磨き直す。本 ADR ではあくまで「方向性」を固定する
- 抽象度の問題は、不変項の各 Level に「最低限の条件 + 書き方の規約（DSL）」を併設することで緩和する
- 「リファレンス実装を作らない」方針は再評価する余地を残す（必要となった時点で別 ADR を起こす）

## 議論ログ（Discussion log）

時系列で、壁打ちで何が見えてきたかを要約。

### Step 1: 「壁打ちで全部見直す必要があるか？」

ユーザーから: 「いちど、プロダクトのイメージを壁打ちして全部みなおしたほうがいいかい？」
私の判断: Yes。23 個の Issue を投入する手前で核を確認すべき。
論点を 7 つ提示: 本体 / 対象ユーザー / 使われ方 / 差別化 / 収益 / 5 年後 / 自立の定義

### Step 2: 論点の選別

ユーザーの判断:
- 本体は最後に
- 対象ユーザーは無視（実態は yuki yamada さん自身）
- 使われ方: 「私が使う。開発させる」
- 差別化は無視
- 収益は無視
- 5 年後: 「これがスタンダードか、類似の大手が作ったものになる」
- 自立の定義が間違ってる気がする

### Step 3: 「自立」の解釈候補

私から 3 候補を提示：A（組織が自走、現Vision）/ B（個人の作業を AI が遂行）/ C（判断軸の自立、メタ自立）。

ユーザー: 「A だけど A じゃない。承認が必要なものだけ承認。これは今の君と僕の関係に似ている」

→ **境界の事前定義 + 内側は委任**が本質と判明。

### Step 4: 本体の再定義（v2）

私から: 「ai-org-os = 承認境界と組織記憶を Git で表現する、境界付き委任のフレームワーク」を叩きとして提示。

ユーザー: 「骨格そのものは本質じゃない。骨格を定義できる、が本質。毎回 Claude なりで skill つくったり、ハーネスしたりするじゃん？ そのフレームワークになる」

→ **骨格を定義するためのメタフレームワーク**であることが判明。Claude Code の組織版という比喩が成立。

### Step 5: 「例はいらない」「外せない部分」

ユーザー: 「例はいまいらない。骨格は組織によって変わるし成長する。だけど、外せない部分。これはダメです。かえられません。という定義の部分を最初に定義してあげる。つまり『開発チームってなに？』」

→ **不変項（invariant）アプローチ**が確定。「開発チームとは何か」のオントロジー定義が本体。

### Step 6: 思考主体への分解

私から「人 or AI が存在」という Level 0 を提示。

ユーザー: 「人はいらん。人はあくまで思考と物理デバイス。組織としての特定の思考とかだとおもうよ。絶対に人はいらない」

→ **組織内に「人間」というカテゴリは存在しない**ことが確定。組織は純粋な思考のネットワーク。

### Step 7: 到達

ユーザー: 「きてる。きてるね、この方向だ」

→ 本 ADR の方向で固定。既存成果物を `OLD/` へアーカイブし、不変項フレームワークの再構築を始める。

## 次にやること（参考、別途 Issue 化予定）

1. 不変項の試案（Level 0〜4）を練り直す
2. 各不変項の「最低限の条件」を明文化する
