# ADR-0002: 用語と「メタのメタ」構造の確定

> 想定読者: ai-org-os の本体構造を理解・実装・拡張する全員。

## Status

**Accepted** — 2026-05-22

## Context（背景）

[ADR-0001](./0001-ai-org-os-as-invariant-framework.md) で「ai-org-os = 開発組織の不変項を定義するフレームワーク」と方向性を固定した。続く壁打ちセッションで、フレームワーク本体（=メタのメタ側）の構造を詰めた。

詰めの主な軸：
- 組織モデリングの構成要素は何か
- 階層構造はどうなるか
- 各レイヤーの責務分離
- 持続性・自律性の実現方法
- 通信基盤
- 用語の確定

本 ADR は、その壁打ちの到達点と確定事項を SSOT として記録する。

## Decision（決定）

### 1. 組織の First Principle

> **組織 = 共通の目的をもった思考の集合体。**

「目的」は構成要素ではなく**定義の一部**。組織として存在する以上、目的は前提に内包されている。

### 2. 不変項（4要素）

| # | 要素 | 説明 |
|---|---|---|
| 1 | **組織⇔外の境界** | 目的を共有するか否かのライン（組織のアイデンティティ） |
| 2 | **思考⇔思考の境界** | 役割分担 + 記憶共有プロセスの境界 |
| 3 | **思考ごとの記憶** | 私有・不可侵。思考と共に消える |
| 4 | **インターフェース** | 組織または思考が外部と相互作用する窓口 |

「目的」は組織の定義に内包、別要素にしない。「共有記憶」は別要素ではなく、要素 2 の境界を越えるプロセスで形成される派生物。

### 3. 階層構造

```
Realm（ai-org-os のルール適用範囲、実コンテナ）
├─ Warden（独自ルールの解釈・適用・進化を担う思考）
├─ Nexus（MCP サーバー、思考間通信 + 外部 I/F）
├─ Guild A（論理セグメント、コンテナなし）
│   ├─ Guildmaster A（組織運営の思考）
│   ├─ Mind A1（メンバー思考）
│   └─ Mind A2
└─ Guild B
    ├─ Guildmaster B
    └─ Mind B1
```

### 4. 用語（仮想空間メタファーで統一）

| 概念 | 用語 | 説明 |
|---|---|---|
| メタ世界コンテナ | **Realm** | ai-org-os のルール適用範囲、実コンテナ |
| 世界 Claude | **Warden** | Realm の守護者、独自ルールの運営 |
| 組織枠 | **Guild** | 目的を共有する Mind の集合体、論理セグメントのみ |
| マザー Claude | **Guildmaster** | Guild を運営する思考 |
| 思考 Claude | **Mind** | 思考そのもの、コンテナで動く |
| MCP サーバー | **Nexus** | 思考間通信と外部 I/F を担う結節点 |
| 私有記憶 | **Mindspace** | Mind 個別の領域、不可侵 |
| 不変項 | **Axiom** | 開発組織の公理（変えてはならない部分） |
| 明示プロセス | **Dispatch** | Mind 間の明示的通信動作 |

別案として技術寄り（Cluster / Steward / Cell / Shepherd / Agent / Hub / Memory / Invariant / Message）も検討したが、議論全体の文脈（=ai-org-os は仮想空間）との整合性で仮想空間メタファーを採用。

### 5. 各 Claude の責務

| Claude | 主な責務 |
|---|---|
| **Warden** | Axiom の保証、ルール解釈、Guild の生成/破棄、Guild 間調停、**リソース管理（停止判断・回復）** |
| **Guildmaster** | Guild 目的の維持、Mind 管理、Guild 内境界の設定、記憶共有プロセス仲介 |
| **Mind** | 個別実務、Mindspace の管理、Dispatch 経由の他 Mind との通信 |

### 6. ライフサイクル（Mind の生成・破棄）

3段階プロセス：

1. **要求**（Mind または Guild） — 起票
2. **承認**（Guildmaster） — 組織方針・必要性で判断
3. **実行**（Warden） — 実際の生成・破棄、リソース管理

Guild の生成・破棄も同様の3段階だが、要求は外部（人間 / 他 Guild）、承認は Warden、実行も Warden。

破棄時は対象 Mind の Mindspace も同時に消去される（不可侵原則 + 思考と共に消える）。

### 7. 稼働モデル

- **24/365 持続稼働**。Mind は呼び出しを待つ受動的存在ではなく、能動的に動ける主体
- **ウェイク条件なし**: Mind は自由に動く（イベントドリブン制限を入れない）
- **リソース制約は Realm レイヤーで担保**: 例「1H 800M tokens 超過で停止」など。Warden が裏側で管理
- **Mind 側からは制限が見えない**: 物理法則のように振る舞う（重力を意識せず歩く人間と同型）

### 8. 通信基盤（MCP）

Nexus は MCP サーバーとして実装。MCP の3要素を ai-org-os にマッピング：

| MCP | ai-org-os |
|---|---|
| **Resources** | Mind が読む情報源（共有記憶、外部データ、Guild 状態） |
| **Tools** | Mind が実行する操作（Dispatch、外部呼出、生成/破棄要求） |
| **Prompts** | 規範テンプレート（Axiom、共通判断軸） |

「読む / 動かす / 従う」に対応。

Nexus は Warden とは**プロセス分離**する（兼任しない）：
- Warden = 判断、Nexus = 経路
- 障害分離（Nexus 停止 ≠ Warden 停止）
- Nexus はスケール可能

### 9. 記憶モデル

- **Mindspace は私有・不可侵**: 他 Mind は勝手に読めない / 書けない
- **思考が消えれば Mindspace も消える**（人と同じ）
- **共有は Dispatch 経由**: 暗黙的に伝わらず、明示的プロセスを踏む
- 共有された記憶は「組織記憶」として派生的に形成される（Guild の memory 領域）

### 10. サブエージェントではなく仮想空間が必要な理由

ai-org-os は Claude のサブエージェント機能では実装できない。理由：

| 観点 | サブエージェント | Realm |
|---|---|---|
| 存在 | 呼ばれた時だけ動いて消える | 永続的に動く |
| 能動性 | 受動的（親が呼ぶ） | 能動的（自発的に動く） |
| ルール | Anthropic / 親のルール下 | 独自ルール（Axiom）下 |
| 通信 | 親経由のみ | 直接通信、Dispatch 経由 |
| 記憶 | コンテキスト独立、プロセス未規定 | Mindspace の不可侵原則がルール化 |

「呼び出し可能な処理単位」と「住人として存在する主体」は質が違う。

ai-org-os ≒ **AI 思考のための仮想空間（virtual world）**。Minecraft サーバーやメタバースに近い構造を、AI 開発組織の目的に特化させたもの。

## Consequences（影響）

### ポジティブ

- **用語が確定** → 議論・実装・ADR で一貫して使える
- **階層構造が固まる** → 実装の設計が直接書ける（コンテナ構成、責務分離）
- **MCP 採用** → 独自プロトコル不要、Claude ネイティブ、痕跡自動保存、認可機構標準
- **境界付き委任の実現** → Mind は自由、Realm が物理を担う、現実の組織と同型
- **「組織」概念がオントロジー的に整理** → 派生概念（目的・役割・共有記憶）が明確

### ネガティブ

- **MCP・コンテナ知識を前提とする** → 受領者の技術要件が上がる
- **3レイヤーの Claude を常時起動** → コスト懸念（リソース管理で対応する想定）
- **仮想空間メタファーは独自感が強い** → 既存の SaaS/PaaS 概念とのマッピング説明が要る

### リスク

- Nexus が単一点障害（SPOF）になる可能性 → 冗長化設計が必要
- Mind の能動性が制御不能になるリスク → Warden のリソース制限と Axiom 強制で対応
- 用語が新規すぎて他者に通じない可能性 → 用語集の整備で対応

## 議論ログ（Discussion log）

時系列で、本 ADR に至る壁打ちのキーステップ。

### Step 1: 「思考」の定義スキップ
ユーザー: 「報酬に対する活動全般」「思考自体は claude に任せるから skip でいい」
→ 思考の中身は実装（Claude）に委ねる。ai-org-os は「組織レイヤー」を扱う。

### Step 2: 「組織のモデリングをしてる」
私が思考の中身に深入りしようとした → ユーザーが「余計なこと考えてる」と補正
→ 組織モデリング**だけ**に集中する。

### Step 3: 組織の構成要素
ユーザー: 「組織⇔外の境界、思考⇔思考の境界、思考ごとの記憶」
→ 「目的」「メンバー」「役割関係」は構成要素から外れる。

### Step 4: 目的は定義の一部
ユーザー: 「組織がそもそも共通の目的をもった思考の集合体だからね」
→ 「組織 = 共通の目的をもった思考の集合体」が First Principle。

### Step 5: 記憶の不可侵原則
ユーザー: 「思考A が消えたら思考A の記憶は消去する必要がある。不可侵。共有したい場合は共有するプロセスを踏む」
→ Mindspace は私有・不可侵。共有は Dispatch 経由。人と同型。

### Step 6: インターフェースの追加
ユーザー: 「組織や思考が外部と作用するためのインターフェースは必要」
→ 4 要素目に「インターフェース」を追加。

### Step 7: 実装イメージ（コンテナ）
ユーザー: 「組織や思考はコンテナで再現できる。思考コンテナの中でbashでもなんでもいいけど、claude がうごいてればOK」
→ Mind = コンテナ。中で Claude が動く。実装手段は自由。

### Step 8: 組織はコンテナ不要
ユーザー: 「組織をコンテナにすると、外界との接触で不都合が出る」
→ Guild は論理セグメントのみ。実体コンテナを持たない。

### Step 9: メタ世界コンテナと Warden
ユーザー: 「メタ世界のコンテナが一枚必要。中で組織枠 + メタ用のマザーClaude + 思考claude」
→ Realm = 実コンテナ、Warden = 必須。

### Step 10: メタ世界 = ルール適用範囲
ユーザー: 「メタ世界コンテナは外部との接続点じゃなくて、ルールの適用範囲」
→ ゲートウェイではない。分散接続が前提。

### Step 11: 24/365 稼働
ユーザー: 「2 4H 3 6 5D 稼働、こっち側からは issue だけ入れる想定」
→ 仮想空間アプローチ確定。サブエージェントでは不可。

### Step 12: 無制限自由 + Realm 制約
ユーザー: 「基本は無制限。世界ルールで制限をかける。思考側から見たら制限されているように見えない」
→ Mind は物理法則の中で生きる感覚。Warden が裏で管理。

### Step 13: 3段階ライフサイクル
ユーザー: 「組織または組織の思考が要求し、マザーが承認し、世界に依頼する」
→ Mind の生成/破棄は「要求→承認→実行」の3段階。

### Step 14: MCP 採用
ユーザー: 「MCPがいいんじゃないかなぁと。世界Claude か世界が提供する共通のMCPサーバー」
→ Nexus = MCP サーバー。Warden とプロセス分離。

### Step 15: 用語確定
私の提案する仮想空間メタファーで採用。

## 次にやること

本 ADR で「メタのメタ側の最低限」が固まった。次は具体実装に降りる：

1. **Realm の最小実装スケッチ** — Docker Compose / K8s manifest / 最小起動構成
2. **Nexus（MCP サーバー）の最小プロトコル** — 提供する Resources / Tools / Prompts の最小セット
3. **Warden / Guildmaster / Mind の最小規約（CLAUDE.md 相当）** — 各レイヤーの行動規範

5. **Issue 投入のインターフェース** — 人間からの主入力経路
