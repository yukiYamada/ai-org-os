# PR Self-Review Checklist

> 想定読者: PR を出そうとしている全 Mind（=コミットして push しようとしている自分）。

PR を出す前に、**Reviewer Persona の視点**で自分の変更を 1 巡レビューする。
これは「Implementer 視点だけで自己完結する」癖を抑える機械化された安全装置。

## 必須チェック（外部レビュー前に必ず自答）

### 1. 冪等性

- [ ] **同じ操作を 2 回呼んでも安全か？**
  - 特に MCP / RPC / HTTP / メッセージング経由のものは必須
  - リトライ（transport timeout、ネットワーク断、クライアント再起動）で同じ呼び出しが二度走る前提
  - 重複呼び出しが「2 回目だけエラー」になる設計は壊れている
- [ ] 副作用を伴う操作（書き込み・削除・移動）に対して、再実行テストを書いたか？

### 2. 環境依存

- [ ] **外部コマンド・パス・OS を前提にしていないか？**
  - `python3` / `docker` / `git` 等のコマンドが PATH にあるか、`command -v` で検証する
  - パス区切りやエンコーディングの違い（Windows / macOS / Linux）を踏まえているか
- [ ] **検証していないか、できないなら警告 or 早期失敗にしているか？**
  - 「自分の環境で動いた」≠「他環境で動く」

### 3. ドキュメント整合性

- [ ] **README / docstring / コメントに書いた手順を**全部**試したか？**
  - 「使い方」を 3 通り書いたなら 3 通りとも動かす
  - import path、起動コマンド、引数の組み合わせ、すべて
- [ ] 古い前提が残っていないか？（Phase の名称、廃止された関数名、移動済みファイルパス）

### 4. 境界条件

- [ ] **empty / max / 重複 / 既に処理済み / 存在しない**、それぞれの挙動を確認したか？
  - 空入力、最大長、特殊文字、None
  - 既に削除されたものを削除する、既に ack されたものを ack する
  - 存在しないリソースへのアクセス
- [ ] 入力検証は **boundary（外部入力）でやっているか、内部 trust を仮定しているか**？

### 5. エラー伝播

- [ ] **失敗時の戻り値が呼び出し側で扱える形か？**
  - `{ok: false, error: "..."}` のような構造化された戻り値か、それとも例外を投げるか、ポリシーが一貫しているか
- [ ] エラーメッセージは「次に何をすればいいか」を含んでいるか？
- [ ] **組み込み例外（`PermissionError` / `OSError` / `ValueError` 等）と意味的に被るドメインエラーは独自クラスを定義したか？**
  - 例: ai-org-os の認可拒否は `AuthorizationError`（独自）。built-in `PermissionError` を再利用すると、fs 操作が出す OS-level の同名例外と区別できず、「**インフラ障害を認可拒否として誤報告**」する事故が起きる（Codex P2 PR #27 follow-up）
  - 独自例外は **built-in を継承しない**こと（`isinstance(e, PermissionError)` が False になる必要がある）

## Reviewer Persona を読む

外部レビュアー（人間 / 別 Mind / Codex 等）に出す前に、`runtime/personas/reviewer.md` を読み返す。
そこに書かれている「思考の癖」を自分の変更に当てはめると、見落としに気づきやすい。

特に：
- リスクを 1 つ以上必ず挙げる
- 必須修正と任意改善を分ける
- 根拠を必ず添える

## なぜこのチェックリストがあるか

このチェックリストは **2026-05-22 の Codex 指摘** をきっかけに生まれた。
3 件すべてが「Implementer 視点だけで自己完結し、Reviewer 視点が薄かった」ことに由来していた：

- 冪等性: ack_dispatch が「正常系 + バリデーション」しかテストされていなかった
- 環境依存: spawn-mind.sh が python の存在を検証せず .mcp.json を書いていた
- ドキュメント整合性: docstring に書いた呼び出しパスが実は動かなかった

これらは PR を出す前に Reviewer 視点で 1 巡していれば全部検出できたはずのもの。
**「自分一人で書いて自分でレビュー」は構造的に Implementer 偏重になる**。
このチェックリストはその偏りを機械化して打ち消す。

## 運用上の注意

- このチェックリストはコピペで PR description に貼り付けられる形を維持する
- 項目を増やすときは、**実際に検出漏れがあったケースを根拠として併記する**
- 「形骸化したチェックリスト」を避けるため、半年に一度は項目を見直して、未使用のものは削る
