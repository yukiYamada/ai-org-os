# FAQ

> 想定読者: 初めてのリポジトリ来訪者、導入検討者。短時間で「自分に必要なツールか」を判断したい人。

回答は1〜3文を目安に短くまとめています。詳細は各リンク先を参照してください。

## Q1. これは何？

**ai-org-os は、メンテナンス性の高い自社開発 AI チームを構築・共有可能にするツールです。** Git リポジトリ上の運営構造（規約・役割・記憶）として AI 開発組織を表現し、立ち上げ・継続運用・他社への移植を支援します。

## Q2. 誰が使うべき？

自社開発を AI 駆動で加速したい事業会社の意思決定者・リード・テックリードが第一義の対象です。第二義として、AI 開発組織を他者へ提供するコンサル・エージェンシー・社内プラットフォームチームを想定しています。

## Q3. 使わない方がいい場面は？

「動く AI を作りたい個人」「単発の AI コード生成だけで足りるケース」「実行ランタイム（オーケストレータ・課金・UI）が今すぐ必要なケース」は対象外です。本ツールは継続運用される組織構造を作るためのもので、ランタイムは現フェーズの非ゴールです。

## Q4. LLM ラッパーやコード生成ツールと何が違う？

ai-org-os は LLM を呼び出しません。代わりに**組織運営構造そのもの**（役割・規約・学習ループ・移植可能なバンドル）を Git で表現します。単発のコード生成ではなく、継続的に自己改善する開発チームを構築・共有することが目的です。

## Q5. 無料？ライセンスは？

現時点では `LICENSE` がプレースホルダー状態であり、private-first で運用しています。外部配布・再利用の条件は確定後に明記します。検討中の段階では、本リポジトリの内容を社外配布する前に必ず最新の `LICENSE` を確認してください。

## Q6. どれくらいで導入できる？

README の [Quick Start](../README.md#quick-start3分で試す) は3分でバンドル概要を把握し、その後 `install.md` の5ステップで自社リポジトリへ展開する流れです。Must レベルのカスタマイズまで含めて、初回適用の目安は半日〜1日です。

## Q7. どの規模の組織向け？

スイートスポットは**小〜中規模チーム（数名〜数十名）**です。Git と Markdown ベースの軽量構造のため、大企業の制度的な承認フローよりも、判断速度を重視する組織との相性が高くなります。

## Q8. 既存の開発プロセスと衝突しない？

ai-org-os は **Vision-First（ビジョンを SSOT とする）** と **人間による最終承認** を原則としています。既存プロセスを置換するのではなく、AI チームの振る舞いを既存プロセスに整合させる枠組みです。承認・説明責任・実行環境の提供は人間側に残ります。

## Q9. AI エージェントは必須？人間チームでも使える？

AI エージェントは必須ではありません。役割・規約・学習ループの構造は人間チームにも適用可能であり、まず人間で運用してから AI エージェントを段階的に組み込む使い方も想定しています。

## Q10. カスタマイズは必須？

Must レベルのカスタマイズ（自社固有名・ドメイン語彙の置換など）だけで使い始められます。`customize.md` に Must / Should / Could の3段階で明示されており、Must 以外は運用しながら段階的に拡張する設計です。

## Q11. 更新はどう追従する？

バンドルは **snapshot model**（決定 D-0003）で配布されます。つまり、配布時点のスナップショットをコピーして自社リポジトリで運用する方式で、上流の更新は自動同期されません。新バージョンが出た際は差分を確認し、必要分のみ取り込みます。

## Q12. コントリビュートしたい場合は？

貢献手順は [`CONTRIBUTING.md`](../CONTRIBUTING.md) を参照してください。Vision との整合性が最優先の判断軸となります。

## Q13. 質問・相談は？

現状の窓口は GitHub Issues です。Discussion / RFC プロセスやコミュニティ運営は中長期で整備予定です（[Roadmap](./roadmap.md) 参照）。
