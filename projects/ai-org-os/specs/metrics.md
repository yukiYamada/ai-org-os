# Metrics（状態監視）定義（Issue 5）

> 想定読者: Planner / Reviewer / Retrospective Facilitator、および Metrics Review を実施する全員。

## 目的
- [Product Vision](../../../docs/product_vision.md) の **Maintain（継続運用・自己改善）** 軸を担保する観測装置。
- 「良化／悪化」を主観でなく **定量で判断** し、自己改善ループ（要件9）の根拠を提供する。
- 本ファイルが Metrics の SSOT（唯一の情報源）。workflow.md / Issue 本文の記述は将来的に本ファイルへの参照に置き換える対象。

## 5指標の定義

### 1. 自立実行率
- **定義（算出式）**: 追加指示なしで完了した Issue 数 / 全完了 Issue 数
- **計測頻度**: 週次（毎週末）
- **初期しきい値**: 60% 以上
- **悪化検出の判定**: 2週連続で目標未達 → ルール改定候補として `decisions.md` に起票

### 2. 手戻り率
- **定義（算出式）**: レビュー差し戻し Issue 数 / 全レビュー Issue 数
- **計測頻度**: 週次（毎週末）
- **初期しきい値**: 20% 以下
- **悪化検出の判定**: 2週連続で目標未達 → ルール改定候補として `decisions.md` に起票

### 3. 承認待ち時間
- **定義（算出式）**: 人間承認待ちの累積時間（週次合計、時間単位）
- **計測頻度**: 週次（毎週末）
- **初期しきい値**: 前週比 -20%（短縮方向）
- **悪化検出の判定**: 2週連続で前週比改善が見られない → ルール改定候補として `decisions.md` に起票

### 4. 提案採択率
- **定義（算出式）**: 採択された提案数 / 全提案数
- **計測頻度**: 週次（毎週末）
- **初期しきい値**: 30% 以上
- **悪化検出の判定**: 2週連続で目標未達 → ルール改定候補として `decisions.md` に起票

### 5. 学習反映率
- **定義（算出式）**: rules / specs / backlog へ反映された learning / decision 数 / 全 learning / decision 数
- **計測頻度**: 週次（毎週末）
- **初期しきい値**: 50% 以上
- **悪化検出の判定**: 2週連続で目標未達 → ルール改定候補として `decisions.md` に起票

## 週次レビュー運用
1. 毎週末に5指標を記録する（`teams/*/memory/learnings.md` への追記、または専用ログファイル）。
2. 目標未達の指標は次サイクル `backlog.md` に改善タスクとして起票する。記載項目:「症状 / 原因仮説 / 最小変更案 / 影響範囲」。
3. 2週連続で未達となった指標は、ルール改定候補として `decisions.md` に上げ、ルール本文（rules.md / specs）の修正を検討する。
4. 翌週レビューで「改善前後の指標差分」を確認し、未改善なら追加で再分解する（workflow.md の「自己改善ループ」と連動）。

## 記録テンプレート（最小）

```
## YYYY-MM-DD 週次 Metrics
- 自立実行率: X/Y = NN%（目標 60%、判定: ◯/△/✕）
- 手戻り率: X/Y = NN%（目標 20%以下、判定: ◯/△/✕）
- 承認待ち時間: NN 時間（前週比 ±NN%、目標 -20%、判定: ◯/△/✕）
- 提案採択率: X/Y = NN%（目標 30%、判定: ◯/△/✕）
- 学習反映率: X/Y = NN%（目標 50%、判定: ◯/△/✕）
- 未達指標と改善タスク:
  - <指標名>: <症状> / <原因仮説> / <最小変更案> / <影響範囲>
```

## 自動化の指針（現状はやらない）
- **非ゴール**: 自動ダッシュボード、外部監視基盤、リアルタイム集計。
- 当面は **手計測で2〜3週運用** し、指標の妥当性（取得可能性・有意性）を検証してから自動化を検討する。
- 手計測で運用が回らないことが確認できた時点で、自動化を別 Issue として起票する。
