# マージ後整合性チェック（全体確認 / 2026-05-21）

## 1. 目的
- 依頼内容「**全体として不整合が起きていないか**」に対し、静的チェックだけでなく、
  **ドキュメント間の意味的整合性**まで含めて確認する。
- 変更量を抑えるため、まず「検出→最小修正→残課題の分離」の順で進める。

## 2. 実施範囲
- `org/`
- `L2_product-core-os/org/`
- `teams/web-service-team/`
- `projects/ai-org-os/`

## 3. チェック観点（全体）
1. 構文整合: コンフリクトマーカー残骸
2. 参照整合: Markdown相対リンク
3. 単一情報源（SSOT）整合: 移行済みファイルの重複定義有無
4. 役割契約整合: ロール契約の入力参照先と現行配置の一致
5. 運用整合: ワークフロー記述と現行リポジトリ構造の齟齬

## 4. 検出結果（重要度付き）

### Critical（修正済み）
- **C-01: `global_rules` の二重定義リスク**
  - 状況: `org/global_rules.md` は「移行案内」と書かれている一方で、本文ルールを保持しており、
    `L2_product-core-os/org/global_rules.md` と実質二重管理になっていた。
  - リスク: 片側のみ更新されると規約矛盾が発生する。
  - 対応: `org/global_rules.md` を互換スタブ（参照案内のみ）へ最小修正。

### High（修正済み）
- **H-01: ロール契約の参照先が旧パス**
  - 状況: `Planner` 入力が `org/global_rules.md` を直接参照していた。
  - 対応: `L2_product-core-os/org/global_rules.md` 参照へ更新。

- **H-02: ロール契約の入力パスが抽象的すぎる**
  - 状況: Builder 入力が `specs/` と `rules/` で、実配置と一致しない。
  - 対応: `projects/*/specs/` と `teams/*/rules.md` へ明示。

### Medium（未修正 / 次タスク化）
- **M-01: ワークフローの「Engineer が `src/` 実装」記述と現状の乖離**
  - 現状: 本リポジトリは documentation-first 段階で、`src/` 前提は将来想定。
  - 提言: 現段階の運用注記（「現フェーズでは主成果物は docs」）を追記すると誤解が減る。

## 5. 実施した最小修正
1. `org/global_rules.md` を案内専用スタブへ整理（SSOTをL2へ一本化）
2. `teams/web-service-team/roles/planner_builder_reviewer_contract.md` の入力参照先を現行配置へ合わせて修正
